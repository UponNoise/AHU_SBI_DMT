# 数据库课程重要知识点整理 (L3-L17)

## L03-SingleEntity-Basics (单实体基础)

### 1. 关系模型基础 (Relational Model Basics)
- **概念**：关系模型是现代数据库系统的基础，由Codd在1970年提出，基于数学集合理论
- **组成**：由关系(表)、属性(列)、元组(行)组成
- **特点**：所有行存储相同类型的项目，每行有唯一标识符

### 2. 实体与属性 (Entities and Attributes)
- **实体**：对应应用领域中的具体事物，用矩形表示，实例对应表中的行
- **属性**：描述实体的离散数据元素，应具有有意义的名称
- **示例**：
  ```
  Shares (实体)
  - code (属性)
  - name 
  - price
  - quantity
  - dividend
  - pe
  ```

### 3. 标识符/主键 (Identifiers/Primary Keys)
- **作用**：唯一标识实体实例
- **最佳实践**：使用与领域属性无关的标识符(如ID)保证唯一性
- **表示法**：星号(*)或"PK"标记
- **示例**：
  ```
  Shares
  *ID code name price quantity dividend pe
  ```

### 4. 命名约定 (Naming Conventions)
- **表名**：复数形式，Upper Camel Case(如OrganizationNames)
- **列名**：单数形式，Lower Camel Case(如organizationName)
- **主键字段**：统一使用ID
- **避免**：缩写和首字母缩略词(除非广为人知)

### 5. SQL数据类型 (SQL Data Types)
- **数值类型**：integer, smallint, float(p), decimal(p,q)
- **字符串类型**：char(n), varchar(n), text
- **日期/时间类型**：date, time, timestamp, time with time zone

## L04-SingleEntity-Other (单实体进阶)

### 1. WHERE子句谓词 (WHERE Clause Predicates)
- **基本比较**：=, <>, >, >=, <, <=
- **IN/NOT IN**：值是否在集合中
  ```sql
  SELECT * FROM Shares WHERE code IN ('FC', 'AR', 'SLG');
  ```
- **BETWEEN**：范围查询
- **LIKE**：模式匹配(使用*或%作为通配符)
- **IS NULL/IS NOT NULL**：空值检查

### 2. 排序输出 (Ordering Output)
- **ORDER BY**：指定排序方式(ASC升序默认，DESC降序)
  ```sql
  SELECT * FROM Shares WHERE pe >= 12 ORDER BY pe DESC, shareName;
  ```

### 3. 派生数据 (Derived Data)
- 计算列可作为查询结果
  ```sql
  SELECT shareName, price, quantity, (dividend/price)*100 AS yield FROM Shares;
  ```

### 4. 内置函数 (Built-in Functions)
- **聚合函数**：COUNT, AVG, SUM, MIN, MAX
- **示例**：
  ```sql
  SELECT AVG(dividend) AS averageDividend FROM Shares;
  ```

### 5. GROUP BY分组
- 将具有共同值的行投影到较小的行集
- 常与聚合函数配合使用
  ```sql
  SELECT sponsorName, AVG(budget) AS averageBudget 
  FROM Awards GROUP BY sponsorName;
  ```

## L06-OneToMany (一对多关系)

### 1. 数据库设计图 (DB Design Diagrams)
- **ERD(实体关系图)**：高级抽象，独立于DB实现
- **RDM(关系数据模型)**：假设关系型实现，处理表和关系

### 2. 一对多关系 (One-to-Many Relationship)
- **表示法**：使用"乌鸦脚"表示法(多端显示乌鸦脚)
- **实现**：通过主键/外键对实现
- **示例**：
  ```
  Nations (1) —— (多) Shares
  Shares表包含nationID外键指向Nations表的ID主键
  ```

### 3. 参照完整性约束 (Referential Integrity Constraint)
- **规则**：每个外键值必须存在对应的主键值
- **要求**：
  - 外键不能为null
  - 必须先创建主键才能定义外键

### 4. Access中创建关系
- 步骤：数据库工具→关系→创建新关系→选择表和列→强制参照完整性

## L07-MultiTableQueries (多表查询)

### 1. 多表查询基础
- **步骤**：
  1. 连接表形成新表
  2. 在连接表上执行SQL查询

### 2. 连接操作 (JOIN)
- **显式连接**：使用INNER JOIN语法
  ```sql
  FROM Nations INNER JOIN Shares ON Nations.ID = Shares.nationID
  ```
- **隐式连接**：在WHERE子句中指定连接条件
  ```sql
  FROM Shares, Nations WHERE Shares.nationID = Nations.ID
  ```

### 3. HAVING子句
- 对GROUP BY结果进行过滤
- 必须与GROUP BY一起使用
  ```sql
  GROUP BY Nations.nationName HAVING Count(*)>=2;
  ```

### 4. 子查询 (Subqueries)
- 查询嵌套在另一个查询中
- **示例**：报告所有澳大利亚股票
  ```sql
  SELECT shareName FROM Shares
  WHERE Shares.nationID IN (SELECT ID FROM Nations WHERE nationName = 'Australia');
  ```

## L09-ManyToMany (多对多关系)

### 1. 多对多关系建模
- **问题**：无法直接在关系数据库中表示
- **解决方案**：引入关联实体(中间表)转换为两个一对多关系

### 2. 关联实体 (Associative Entity)
- **特点**：
  - 包含两个外键(分别指向两个实体)
  - 可包含关系特有属性(如数量、价格)
- **示例**：
  ```
  Sales (1) —— (多) LineItems (多) —— (1) Items
  ```

### 3. 三表连接查询
- **示例**：列出销售日期、商品名称和数量
  ```sql
  SELECT saleNo, saleDate, itemName, quantity 
  FROM Sales, LineItems, Items
  WHERE Sales.ID = LineItems.saleID AND Items.ID = LineItems.itemID;
  ```

### 4. EXISTS/NOT EXISTS
- **EXISTS**：子查询返回至少一行时返回true
- **NOT EXISTS**：子查询不返回任何行时返回true
  ```sql
  SELECT itemName FROM Items 
  WHERE itemtype = 'C' AND EXISTS (SELECT * FROM LineItems WHERE LineItems.itemID = Items.ID);
  ```

## L10-OneToOne (一对一和递归关系)

### 1. 一对一关系 (1:1 Relationship)
- **实现选择**：外键可放在任一表中
- **示例**：部门与部门主管的关系
  ```
  Departments表包含deptHeadID外键指向Employees表
  ```

### 2. 递归关系 (Recursive Relationship)
- **定义**：实体与自身的关系
- **示例**：员工与上级的关系
  ```
  Employees表包含bossID外键指向同一表的ID主键
  ```

### 3. 递归关系查询
- **技巧**：为同一表创建别名
  ```sql
  SELECT E.firstName AS Employee, B.firstName AS Boss
  FROM Employees E, Employees B
  WHERE E.bossID = B.ID;
  ```

## L11-Normalization (规范化)

### 1. 规范化目的
- 消除冗余数据
- 避免插入、更新和删除异常
- 使数据修改只需在一个表中进行

### 2. 函数依赖 (Functional Dependency)
- **定义**：一个或一组属性决定另一个属性的值
- **表示法**：A → B (A决定B)
- **完全函数依赖**：需要所有属性才能确定另一个属性

### 3. 范式 (Normal Forms)
- **1NF**：
  - 所有行具有相同数量的列
  - 只包含单值属性
- **2NF**：
  - 已是1NF
  - 非主键列必须完全依赖于整个主键(主要针对复合主键)
- **3NF**：
  - 已是2NF
  - 非主键列不能依赖于其他非主键列(即无传递依赖)

### 4. 更新异常 (Update Anomalies)
- **插入异常**：无法插入没有员工的分销中心信息
- **删除异常**：删除最后一个员工会意外删除分销中心信息
- **修改异常**：修改分销中心信息需更新多条记录

## L12-Examples (规范化示例)

### 1. 规范化过程示例
- **非规范化表**：
  ```
  StaffDistributionCenter(ID, staffNo, staffName, position, salary, dCenterNo, dAddress, dTelNo)
  ```
- **问题**：dAddress和dTelNo依赖于dCenterNo而非主键ID

### 2. 规范化解决方案
- **分解为两个表**：
  ```
  Staff(ID, staffNo, staffName, position, salary, dCenterID)
  DistributionCenters(ID, dCenterNo, dAddress, dTelNo)
  ```

### 3. 牙医数据库规范化
- **原始表问题**：
  - dentistName依赖于staffNo
  - patName依赖于patNo
  - surgeryNo不依赖于上述任何属性
- **规范化设计**：
  ```
  Dentists(staffNo, firstName, lastName)
  Patients(patNo, firstName, lastName)
  Appointments(ID, staffNo, patNo, apDate, apTime, surgeryNo)
  ```

- **目标**：识别数据库中要存储的数据结构
- **合作过程**：客户与分析师的合作

## L14-Views & Forms (视图和表单)

### 1. 视图 (Views)
- **定义**：派生表(虚拟表)，行为类似基表但不物理存在
- **优点**：
  - 减少数据库变更的影响
  - 简化数据库使用
  - 限制数据访问权限
  - 自定义数据库外观

### 2. 分层表单 (Hierarchical Forms)
- **组成**：主表单和子表单
- **用途**：为1:M关系提供有吸引力的界面
- **类型**：
  - 详情表单(单一数据源)
  - 分割表单(详情表单+数据表)
  - 子表单(父子表单)
  - 多项目表单

### 3. 表单设计步骤
- 选择数据源(表或查询)
- 选择表单类型
- 创建初始表单后可重新排列或删除控件

## L15-Reports (报表)

- **术语**：

### 2. 报表组件
- **报表部分**：
  - 报表页眉/页脚

  - 组页眉/页脚

### 3. 报表向导使用

## L17-DataModeling-Leftovers (数据建模补充)

### 1. 实体类型 (Entity Types)
- **独立实体**：起始点，在客户思维中突出
- **依赖实体**：依赖其他实体存在和标识
- **关联实体**：多对多关系的产物
- **聚合实体**：由多个具有共同前缀/后缀的实体创建
- **从属实体**：数据在不同实例间可变的实体

### 2. 泛化 (Generalization)
- **定义**：更通用元素与更特定元素间的关系
- **映射**：为每个实体创建一个表，子类型实体主键与超类型相同

### 3. 聚合 (Aggregation)
- **定义**：两个实体间的部分-整体关系
- **类型**：
  - 共享聚合：一个实体拥有另一个实体，但其他实体也可拥有
  - 组合聚合：一个实体独占拥有另一个实体

### 4. 数据建模技巧
- 模型会扩展和收缩
- 必要时创建标识符
- 标识符应只有一个目的——标识
- 数据模型不隐含排序
- 创建属性如需排序
- 属性含义必须一致

# 数据库设计与实践关键知识点总结

## 1. 数据库索引 (Database Indexing)
- **B+树索引**: 最广泛使用的索引结构，支持范围查询和等值查询，时间复杂度O(log n)
  - **B+ Tree**: Most widely used index structure supporting range and equality searches with O(log n) time complexity
- **索引类型**: 主键索引、组合索引、多列索引
  - **Index types**: Primary key indexes, composite indexes, multi-column indexes
- **索引规则**: 
  - 为每个主键创建索引
  - 对频繁出现在WHERE子句中的属性创建索引
  - 避免冗余索引
  - **Indexing rules**:
    - Index every primary key
    - Index attributes frequently used in WHERE clauses
    - Avoid redundant indexes

## 2. 关系模型与SQL (Relational Model and SQL)
- **关系模型基础**: 表、元组、属性、域约束、键约束、参照完整性约束
  - **Relational model basics**: Tables, tuples, attributes, domain constraints, key constraints, referential integrity constraints
- **SQL语句分类**:
  - DDL(数据定义语言): CREATE, ALTER, DROP
  - DML(数据操作语言): SELECT, INSERT, UPDATE, DELETE
  - DCL(数据控制语言): GRANT, REVOKE
  - TCL(事务控制语言): COMMIT, ROLLBACK
  - **SQL categories**:
    - DDL (Data Definition Language)
    - DML (Data Manipulation Language) 
    - DCL (Data Control Language)
    - TCL (Transaction Control Language)

## 3. 复杂查询 (Complex Queries)
- **公用表表达式(CTE)**: WITH子句定义的临时命名结果集
  - **Common Table Expressions (CTE)**: Temporary named result sets defined in WITH clauses
- **递归查询**: 处理层次结构数据(如组织结构、零件爆炸图)
  - **Recursive queries**: For hierarchical data (org charts, bill of materials)
- **OLAP查询**: ROLLUP, CUBE, GROUPING SETS用于多维数据分析
  - **OLAP queries**: ROLLUP, CUBE, GROUPING SETS for multi-dimensional analysis
- **窗口函数**: OVER, PARTITION BY用于计算移动平均值、排名等
  - **Window functions**: OVER, PARTITION BY for moving averages, rankings etc.

## 4. 逻辑数据库设计 (Logical Database Design)
- **ER模型**: 实体、关系、属性、基数约束
  - **ER model**: Entities, relationships, attributes, cardinality constraints
- **设计流程**: 需求分析→概念建模→视图集成→关系表转换→规范化
  - **Design process**: Requirements→Conceptual modeling→View integration→Table transformation→Normalization
- **ER到关系表的转换**:
  - 1:1, 1:N关系添加外键
  - M:N关系创建连接表
  - 递归关系使用自引用外键
  - **ER to relational mapping**:
    - 1:1, 1:N relationships add FKs
    - M:N relationships create junction tables
    - Recursive relationships use self-referencing FKs

## 5. 大数据系统 (Big Data Systems)
- **大数据特征**: 体量(Volume)、速度(Velocity)、多样性(Variety)
  - **Big data characteristics**: Volume, Velocity, Variety
- **并行数据库架构**: 共享内存、共享磁盘、无共享
  - **Parallel DBMS architectures**: Shared-memory, shared-disk, shared-nothing
- **NoSQL数据库类型**:
  - 键值存储(Redis, Riak)
  - 文档数据库(MongoDB, CouchDB)
  - 列族存储(Cassandra, HBase)
  - 图数据库(Neo4j)
  - **NoSQL types**:
    - Key-value stores
    - Document stores  
    - Column-family stores
    - Graph databases
- **CAP定理**: 一致性(Consistency)、可用性(Availability)、分区容错性(Partition Tolerance)三者不可兼得
  - **CAP theorem**: Can only achieve 2 of 3: Consistency, Availability, Partition tolerance

## 6. MapReduce
- **编程模型**: Map(映射)和Reduce(归约)函数
  - **Programming model**: Map and Reduce functions
- **Hadoop生态系统**: HDFS(存储)、YARN(资源管理)、MapReduce(处理)
  - **Hadoop ecosystem**: HDFS (storage), YARN (resource mgmt), MapReduce (processing)
- **与并行数据库比较**:
  - 优点: 简单、可扩展、容错、成本低
  - 缺点: 性能通常低于并行数据库
  - **vs Parallel DBMS**:
    - Pros: Simple, scalable, fault-tolerant, low cost
    - Cons: Generally lower performance than parallel DBMS

## 7. 实际应用案例 (Practical Examples)
- **30天再入院查询**: 使用自连接和窗口函数分析医院再入院情况
  - **30-day readmission queries**: Using self-joins and window functions to analyze hospital readmissions
- **病理图像数据库**: 管理高分辨率医学图像及其元数据
  - **Pathology image DB**: Managing high-res medical images and metadata
- **多语言持久化**: 根据数据特点混合使用不同数据库技术
  - **Polyglot persistence**: Using different DB technologies based on data characteristics
